# Clear_Architecture
Практика в Архитектуре на примере управления роботом уборщиком

## Решение в процедурном стиле
Записи определенного типа все еще существуют в функциональном стиле, значит я могу переиспользовать часть классов.

Интерпретатор у меня так и был в функциональном стиле.  
Надо посмотреть образец решения, что бы решить как двигаться дальше.   
В императивном стиле есть просто глобальные переменные в контексте большого скрипта.    
В процедурном стиле могу вынести из объектов методы с ними работающие, если таковые все еще есть.  
Например в Cleaner оставить только поля.  
А методы работы с ними вынести и обращаться к его полям через геттеры, передавая по цепочке преобразованный объект.  

Возвращать из функций буду пару из объекта(record) сообщения и новый объект cleaner, т.к. его поля сделаю immutable.

Методы в классе, куда вынес функции работы с пылесосом сделал статическими.

В первый раз забыл передать переменную в которой будет оказываться ссылка на новый объект, а передал переменную с постоянной ссылкой.
Типичная ошибка для измененния значения в цикле.

## Анализ по решению в любом стиле

Начал с мышления письмом и понял, что могу быстренько выделить основные АТД по Бертрану Мейеру, т.к. это наиболее знакомый мне стиль проектирования.
Ввел ограничения для АТД, например, то что метры должны быть положительные
После движения робот не может оказаться за пределами поля
То что угол должен быть в пределах 360 градусов и внутри программы будет положительным (-90 преобразую в 270 на входе).
Интересно что даже в наивном стиле выделили Enum, это уже не мало :).

IDE открыл только когда реализовал все классы в блокноте. Так не распыляюсь на более богатый интерфейс IDE и возможности запуска кода.
Поправить потребовалось буквально 5-20 строк геттеров и добавить импорты.
Была одна логическая ошибка, когда использовал Map вместо List<Pair<String, Object>>.

Минусы
* Смущает, что приходится выполнять каст типа перед передачей аргумента в метод.

Плюсы
* Саму программу после выделения всех АТД и введения ограничений было уже проще реализовать, т.к. все для этого было.  
* Этим методом проектирования реализовывал и более трудные программы.
