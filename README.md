# Clear_Architecture
Практика в Архитектуре на примере управления роботом уборщиком

## Решение в процедурном стиле
Записи определенного типа все еще существуют в функциональном стиле, значит я могу переиспользовать часть классов.

Интерпретатор у меня так и был в функциональном стиле.  
Надо посмотреть образец решения, что бы решить как двигаться дальше.   
В императивном стиле есть просто глобальные переменные в контексте большого скрипта.    
В процедурном стиле могу вынести из объектов методы с ними работающие, если таковые все еще есть.  
Например в Cleaner оставить только поля.  
А методы работы с ними вынести и обращаться к его полям через геттеры, передавая по цепочке преобразованный объект.  

Возвращать из функций буду пару из объекта(record) сообщения и новый объект cleaner, т.к. его поля сделаю immutable.

Методы в классе, куда вынес функции работы с пылесосом сделал статическими.

В первый раз забыл передать переменную в которой будет оказываться ссылка на новый объект, а передал переменную с постоянной ссылкой.
Типичная ошибка для измененния значения в цикле.

### Плюсы
* Могу отделить данные от процедур их обрабатывающих.
* Тесты менять не пришлось.
* Мутабельность ушла от нескольких полей в одну переменную со ссылкой на объект

### Минусы
* Методы статичные, то есть переопределять их придется на данном этапе либо меняя изначальную реализацию, либо меняя код вызова.
Дополнительную абстракцию в виде функционального интерфейса не подкладывал, да и это наверное ближе к ООП.
* Осталась мутабельная переменная, в которой меняю ссылку на новый объект в каждой итерации цикла
